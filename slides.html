SECTION 02

목표 시스템 다이어그램

8단계 순환 — 우리가 지을 집의 설계도

  • ① 공통 폴더·파일 생애주기 구조 → ② Git 버전관리(공통)
  • ③ 완성 프로젝트 inbox 이동(팀원) → ④ inbox 검토(지식관리자) → ⑤ 지식컴파일
  • ⑥ raw 이동 + wiki 추출 적립 → ⑦ MCP 서버·권한통제 → ⑧ 권한 내 지식활용 → ①로 순환

상세: 교안 index.html §2

역할이 다르면 하는 일이 다릅니다

  • 각 팀원 — 만들고, 커밋하고, 완성되면 inbox로 보낸다
  • 회사 지식관리자 — inbox 검토, 지식컴파일, 권한 통제
  • 모든 사용자 — 일반 LLM에서 권한범위 내 지식을 꺼내 새 지식을 만든다
✓ 8단계 순환도를 보고 "내 역할은 어디부터 어디까지"를 말할 수 있습니다
SECTION 03

Git의 운용

저장 · 커밋 · 푸시 — 세 단어만 잡으면 됩니다

  • 저장 — 내 컴퓨터 파일에 쓰는 것 (Git은 아직 모름)
  • 커밋 — "이 시점을 기록해 둬" 하고 도장을 찍는 것
  • 푸시 — 도장 찍힌 기록을 원격 저장소(GitHub)로 올리는 것

상세: 교안 index.html §3 개념 정립

라이브 데모 — 더미 파일로 첫 커밋

“이 더미 파일 폴더를 git 저장소로 만들고, 파일을 모두 스테이징해서 ‘첫 커밋: 더미 파일 등록’ 메시지로 커밋해줘”
git init
git add .
git commit -m "첫 커밋: 더미 파일 등록"

강사가 준비한 더미 파일을 내려받아 다 같이 실행합니다. AI가 위 프롬프트를 받아 아래 명령을 실행합니다 — CLI에 능숙하다면 직접 실행해도 됩니다. 끝나면 git log --oneline 으로 도장 찍힌 기록을 확인합니다.

라이브 데모 — 브랜치: 원본은 두고 갈래로 실험

“draft-v2 브랜치를 새로 만들어 전환하고, 더미 파일을 고쳐 커밋한 다음 main으로 돌아와 draft-v2를 병합해줘”
git switch -c draft-v2
git switch main
git merge draft-v2

AI가 위 프롬프트를 받아 아래 명령을 실행합니다 — CLI에 능숙하다면 직접 실행해도 됩니다. draft-v2 갈래에서 더미 파일을 고쳐 커밋한 뒤 main으로 돌아와 합칩니다. checkout이 아니라 최신 명령 switch를 씁니다.

실무 3종 세트

  • 커밋메시지 템플릿 — LLM과 함께 만들어 ~/.gitmessage.txt 로 지정, 커밋 때 기본 헤드메시지 선택 + 상세 작성
  • LFS — 대용량 파일은 지정 저장소에, Git 히스토리에는 가벼운 포인터만 로컬로 쌓기
  • 바이너리 관리 — 파일별 커밋메시지 + 폴더 루트 히스토리.md 동시 기록

상세: 교안 index.html §3 커밋 템플릿 · LFS · 바이너리 관리법

✓ 더미 파일로 커밋 2회 + 브랜치 병합 1회를 직접 마쳤고, git log --oneline 에 기록이 보입니다
SECTION 04

업무환경에서의 데이터 생성 시스템

무엇을 어디에 — 구조별 Git 관리 지침

  • GitHub로 — 코드 · 워크플로우 · 템플릿 (용량 작고 배포 효과 큰 것)
  • 로컬 Git으로 — 바이너리 · 로컬 업무 파일
  • 각 회사의 실제 생성·보관 구조 예제를 수강생이 제출해 함께 공통 지침을 만듭니다

상세: 교안 index.html §4 구조별 깃관리 지침

규칙은 강제되어야 규칙입니다

  • 전사 공통 폴더·파일 생성 규칙강제 템플릿 작성

  • raw 폴더 이동 시 · 장기보관 이동 시 = 반드시 강제 순찰 통과

상세: 교안 index.html §4 규칙강제 템플릿 · 순찰자

업무 폴더에서 Claude Code를 열고 이렇게 말합니다: "이 폴더에서 우리 명명 규칙에 어긋난 파일을 찾아 목록으로 보고하고, 수정안을 제안해줘." 보고만 시키고 수정 승인은 사람이 합니다.

✓ 우리 회사 기준으로 "GitHub행 / 로컬 Git행" 분류표와 규칙강제 템플릿 초안이 나왔습니다
SECTION 05

LLM Wiki 구축

위키의 뼈대 — 구조 · 의미 · 관계

  • 강사 예제(깃허브 제공)를 받아 우리 목적에 맞게 재작성하는 법을 익힙니다
  • 바이너리 파일(엑셀·PPT·한글·워드·이미지·캐드 등)은 텍스트로 컴파일해야 위키가 읽습니다
  • 컴파일 스킬은 LLM에게 오픈소스 추천부터 요청 — 교안의 예제 프롬프트 사용

상세: 교안 index.html §5 위키 구조 · 바이너리 컴파일 프롬프트

라이브 데모 — 바이너리를 마크다운으로

“markitdown을 전체 옵션으로 설치하고, 이 폴더의 보고서.docx를 보고서.md 마크다운으로 변환해줘”
pip install 'markitdown[all]'
markitdown 보고서.docx -o 보고서.md

AI가 위 프롬프트를 받아 아래 명령을 실행합니다 — CLI에 능숙하다면 직접 실행해도 됩니다. Word·엑셀·PPT·PDF를 한 도구로 변환합니다. 한글(HWP)은 미지원이라 별도 도구(gethwp 등)를 씁니다 — 상세는 교안 토글에.

라이브 데모 — graphify로 지식그래프 만들기

uv tool install graphifyy
/graphify . --wiki

설치 패키지명은 graphifyy(y가 두 개), 명령은 graphify입니다. 폴더를 통째로 넣으면 지식지도(HTML+JSON+위키)가 나옵니다.

라이브 데모 — Qdrant: 의미로 찾는 검색 창고

“qdrant-client 파이썬 패키지를 설치하고, 서버 없이 로컬 메모리 모드(:memory:)로 잘 동작하는지 확인해줘”
pip install qdrant-client

AI가 위 프롬프트를 받아 아래 명령을 실행합니다 — CLI에 능숙하다면 직접 실행해도 됩니다. 벡터 데이터베이스(뜻이 비슷한 문서를 찾아주는 창고)입니다. 서버 없이 QdrantClient(":memory:") 로컬 모드로 바로 실습합니다. Docker 대시보드 경로는 교안 토글에.

✓ 바이너리 1개가 .md로 변환되고, graphify 결과 graph.html이 브라우저에서 열립니다
SECTION 06 · 더 알아보기

지식화 구조를 GBrain으로 일원화하는 심플 구조

GBrain — 한 도구로 끝내는 선택지

  • 마크다운 노트를 넣으면 스스로 연결되는 지식그래프가 됩니다 (서버·Docker 불필요)
  • 벡터 + 키워드 + 그래프 하이브리드 검색, 출처 인용이 붙은 답변(think)
  • graphify는 "일괄 분석 → 산출물", GBrain은 "에이전트의 지속 메모리" — 용도가 다릅니다

상세: 교안 index.html §5-1 GBrain

라이브 데모 — 2분 만에 두 번째 뇌

“GitHub의 garrytan/gbrain을 bun으로 전역 설치하고, pglite 모드로 초기화한 다음 ‘우리 회사 계약 검토 절차는?’이라고 think로 질문해줘”
bun install -g github:garrytan/gbrain
gbrain init --pglite
gbrain think "우리 회사 계약 검토 절차는?"

AI가 위 프롬프트를 받아 아래 명령을 실행합니다 — CLI에 능숙하다면 직접 실행해도 됩니다. init 후 gbrain import ~/notes/ 로 노트 폴더를 넣고 질문합니다. Windows 안내가 아직 빈약해 실습은 macOS 기준입니다.

✓ gbrain doctor 가 정상이고, think 질문에 출처 인용이 붙은 답이 돌아옵니다

지식관리는 한곳에 쌓는 일이 아닙니다

  • 흐르는 정보는 GBrain이 기억합니다 — 업무일지·회의록·메모 (작업 기억)
  • 확정된 지식은 LLM 위키가 보존합니다 — 기준·매뉴얼·정책 (장기 기억)
  • AI는 MCP로 두 기억을 함께 쓰고, 사람이 검토·승인한 것만 승격됩니다

상세: 교안 index.html §5-1 "GBrain과 LLM 위키의 역할 분리"

SECTION 07

MCP 서버 구축 — 권한 있는 사람만 꺼내 쓰게

위키 검색 MCP 서버 — 설계 3원칙

  • 서버 위치 — 관리자만 접근할 수 있는 곳에 배치
  • 구축은 LLM에게 요청 — 우리가 정할 것은 요구사항과 권한 정책
  • 권한등급 2방식 — (a) 토큰으로 PC별 제어 · (b) Authorization 계정별 인증으로 개별 제어

상세: 교안 index.html §6 MCP 서버 구축

라이브 데모 — 위키 서버를 Claude에 연결

claude mcp add --transport stdio wiki -- python server.py
claude mcp list

연결 후 Claude Code 안에서 /mcp 를 입력해 서버 상태와 인증을 확인합니다. -- 뒤가 실제 실행 명령입니다.

통합 스킬 — 묻는 법을 몰라도 답이 나오게

  • 구조 · 관계 · 의미 질문을 프레임웍별로 따로 묻지 않아도 자동 인식해 답하게 하는 통합 스킬
  • 마지막 단계: 일반 LLM에 연결 — 누구나 권한범위 안에서 자유롭게 활용
  • 여기서 새 지식이 태어나 다시 ①번으로 — 순환 완성
✓ claude mcp list 에 wiki 서버가 보이고, 권한 밖 질문은 거절되는 것을 확인했습니다

정책 숙제 — 2차 지식, 바로 원본에 넣을 것인가?

  • A안 — 추출 지식으로 만든 새 지식을 원본지식에 바로 산입
  • B안 — 정거장을 두고 관리자 통제로 검토 후 재지식화
  • 2차 지식 생성은 AI 특성상 왜곡이 섞일 수 있다 — 원본 오염 방지 장치의 수위는 조직의 정책으로
  • 교안 권장 — 2차 합성지식은 별도 구역에 분리 보관, 원본 승격은 전담 담당자만 (만든 사람 ≠ 올리는 사람)

토론: 우리 회사의 승격 담당자는 누구로, 승격 주기는 어떻게? — 그것이 사람과 AI가 같이 일하는 구조입니다.
